요즘 게임 개발이 재밌어서 퇴근하고 남는시간은 전부 개발을 하고 있다..  
8~9월 출시 목표로 개발을 하고 있는데 뭐 솔직히 출시 목표는 처음 잡아보는거라 잘 될지는 모르겠고 그냥 열심히 하고 있다. 재밌어서 열심히 하게 되더라.

예전에 Unity Asset Store에서 Odin이 반값 할인을 하길래 냅다 사왔다. 너무 비싸서 그저 바라만 봤는데 직접 사용하니 만족도가 높긴 하다.  
오늘도 Odin을 사용하며 인스펙터 창을 꾸미고 있는데 디버깅을 하면서 왜 안되는지 이해가 안되는 것을 찾았다.

```Csharp
[Title("Runtime")]
[ShowInInspector, ReadOnly]
private bool _isInitialized;

[ShowInInspector, ReadOnly]
private bool _inputSingletonExists;

[ShowInInspector, ReadOnly]
private Vector2 _currentMoveInput;
```

이게 문제의 필드였다.  
해당 필드는 Update메서드에서 유저의 입력을 감지해서 실제 Vector2 정보를 캐릭터가 움직일 수 있도록 전달하는 역할을 한다.  
그냥 디버깅 할 겸 Odin에서 제공하는 ShowInInspector를 사용해서 구현해 놨는데, 이게 웬걸 직접 실행하면서 눈으로 확인을하니 미동도 없다?  
근데 또 다른 bool 값 두개는 잘 체크가 된다.

캐릭터가 아주 잘 움직이기에 뭔가해서 로그도 찍어 봤는데 저 값을 아주 잘 갱신되고 있었다. 그렇다면 Odin문제인가 싶어서 [ShowInInspector] 대신 [SerializeField]를 사용해봤더니 아주 잘 갱신이 된다.

뭔 차이인가 싶어서 검색을 해서 찾아봤다.

---

## 1. 문제 상황

- **환경**: Unity + Odin Inspector 사용
- **현상**: `private Vector2` 필드에 `[ShowInInspector]`, `[ReadOnly]`를 붙여 모니터링하려 했으나, 게임 실행 중 플레이어가 움직여도 값이 `(0, 0)`으로 고정됨.
- **특이사항**: 같은 스크립트 내의 `bool` 타입 변수들은 체크 박스가 정상적으로 갱신됨.

---

## 2. 원인 분석

### ① 직렬화(Serialization) vs 리플렉션(Reflection)

- **`[SerializeField]`**: 유니티 엔진의 **공식 직렬화 시스템**에 변수를 등록한다. 유니티는 이 값을 직접 관리하므로, 값이 변하면 즉시 인스펙터를 다시 그리도록(Repaint) 명령한다.
- **`[ShowInInspector]` (Odin)**: 유니티가 모르는 `private` 변수를 **리플렉션** 기술로 억지로 화면에 그려준다. 이는 유니티 엔진의 공식 감지 대상이 아니기에 갱신 우선순위가 낮다.

### ② 데이터의 특성 차이 (State vs Data)

- **Bool (상태)**: `true` 또는 `false`로 한 번 변하면 고정된다. 에디터가 다른 이유(마우스 움직임 등)로 화면을 새로고침할 때 이미 바뀐 상태가 유지되므로 "잘 갱신되는 것처럼" 보이는 것.
- **Vector2 (연속 데이터)**: 실시간으로 수치가 계속 변함.  
  에디터가 이 변화를 실시간으로 감시(Polling)하지 않으면 우리 눈에는 멈춰있는 것처럼 보이게 된다.

---

## 3. 해결책

> [SerializeField]와 [ReadOnly]를 써라!

---

## 4. 그럼 [ShowInInspector] 언제 쓰나??

지금 상황에서는 SerializeField와 비교하면 얘 왜쓰나 쓸모가 없어 보이긴 한다. 하지만 이놈의 진가는 SerializeField가 불가능할 때 발휘된다.

| 상황                      | 추천 속성           | 이유                                    |
| :------------------------ | :------------------ | :-------------------------------------- |
| **단순 변수 모니터링**    | `[SerializeField]`  | 가장 빠르고 정확한 갱신                 |
| **Property (프로퍼티)**   | `[ShowInInspector]` | 유니티 기본 기능은 프로퍼티 노출 불가   |
| **Dictionary, Interface** | `[ShowInInspector]` | 유니티가 직렬화하지 못하는 자료구조     |
| **Static 필드**           | `[ShowInInspector]` | 특정 오브젝트에 속하지 않은 전역 데이터 |

Unity에서 Property와 Dictionary는 인스펙터 창에서 확인할 수 없다. 그럴 때 ShowIninspector를 쓰면 아주 만족감 높게 인스펙터 창을 꾸밀 수 있다. 물론 꾸미는 것 만이 아니라 데이터 관리도 더 용이하다.

---

## 5. 이상하게 쓰지는 말자

아무레도 리플렉션으로 제공하는 기능이다보니 오버헤드가 존재하긴 한다. 그래도 Odin의 개발자들은 똑똑해서 자체 최적화를 해두었기 때문에 별 문제는 없다고 한다. 주의해야 할 것은 프로퍼티와 함께 사용할 때이다.

```Csharp
// ❌ 아주 위험한 사용법 (에디터 렉의 주범)
[ShowInInspector]
public List<Enemy> NearEnemies
{
    get
    {
        // 인스펙터를 띄워놓는 내내 매 프레임 이 무거운 탐색 로직이 실행됨!
        return FindObjectsOfType<Enemy>().Where(e => Vector3.Distance(transform.position, e.transform.position) < 10f).ToList();
    }
}

// ✅ 안전한 사용법
[ShowInInspector]
private int _currentHealth; // 단순한 데이터 확인은 안전함
```

예전 게임회사에 있을 때도 에디터가 좀 느렸던 거 같은데, 오딘이 무거워서 라는 답변을 들었던 기억이 있긴 하다. 그래도 인스펙터가 만족스러울 정도로 예뻐지니 포기할 수 없다 오딘은...
