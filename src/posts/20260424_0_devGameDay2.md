오늘은 적 소환 시스템과, 플레이어 추격 시스템, 그리고 서로 뭉치지 않도록 하는 기능을 만들었다.

- 간단하게 카메라가 플레이어를 따라가는 기능도 만들었는데, 평소 MonoBehaviour로만 움직이던 OOP지향적인 프로그래밍에서는 대충 만들었지만, 이번에는 DOTS를 적용했기 때문에, 조금 다르게 스크립트를 작성했다.
  아직은 데이터 지향적인 사고가 익숙치 않다.

---

## 진행 내용

### 1. 적 생성 이슈 해결

적이 정상적으로 생성되지 않는 문제 발생함.

- 원인: TransformUsageFlags
  - `None` 상태 및 `Dynamic` 처리 관련 문제였음
- 해결:
  - 해당 부분 수정하여 정상적으로 적 생성됨

> ECS에서 **컴포넌트 상태 관리가 매우 중요함을 확인함**

---

### 2. Assembly Definition 의존성 문제

다음과 같은 오류 발생함:

> `Unity.Collections 관련 오류`

- 원인:
  - `Game.Presentation.asmdef`에 `Unity.Collections` 의존성이 누락됨
- 해결:
  - asmdef에 의존성 추가하여 정상 동작 확인함

> **DOTS 환경**에서는 **패키지 단위 의존성 관리가 필수적임**

---

### 3. Physics vs Separation System 선택

적 이동 및 충돌 처리 방식에 대해 두 가지 접근 검토함.

#### Unity Physics

- 장점: 구현 단순함
- 단점: 대량 객체 처리 시 성능 부담 존재함

#### Separation System

- 장점:
  - DOTS 구조에 적합함
  - 커스터마이징 용이함
- 단점:
  - 구현 난이도 높음

> 결론:  
> 대량 적 처리를 고려하여 **Separation System 유지하기로 결정함**

---

### 4. 성능 이슈 (O(n²)) 분석

현재 Separation 로직 구조:

- 모든 적이 서로를 검사하는 구조 → **O(n²)**

> 문제:

- 적 개체 수 증가 시 성능 저하 발생 가능성 높음

> 대응 계획:

- 공간 분할(Spatial Partitioning) 도입 예정임

> 검토 대상:

- Grid 기반 Partition
- Quadtree (확장 시 고려)

---

### 5. 공간 분할 개념 정리

기존 구조:

모든 적 ↔ 모든 적 비교 (N²)

개선 구조:
같은 영역 내 객체만 비교

> 효과:

- 비교 대상 수 감소됨
- 연산량 감소 기대됨

---

### 6. 현재 상태 정리

- 적 생성 정상 동작함
- ECS 구조 기본 형태 확보됨
- 성능 개선 필요함 (추후 작업)

## 회고

오늘은 기능 추가보다는 **문제 해결 및 구조 안정화 중심으로 작업 진행함**

느낀 점:
